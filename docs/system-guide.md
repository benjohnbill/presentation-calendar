## 1. 한 문장 요약

친구들이 각자 가능한 **날짜**를 찍고, 4명이 모이면 그 날 모임(세션)이 자동으로 생기며,
디스코드로 알림이 가는 캘린더.

정기 주기가 없다. "4명이 모이면 그 날이 성사된다"는 그룹의 기존 룰을 그대로 소프트웨어로
옮긴 것이다.

---

## 2. 핵심 흐름 — 2단계 구조 (이게 심장이다)

```
[가능 Available]        날짜만 찍음 (시간 X)
      │  4명 모이면
      ▼
[잠정 성립]             "이 날 뜬다" — 디스코드 알림, 시간표 열림
      │
[참석의사 Commit]       시간표에 자기 가능 시간 입력 (시간 O)
      │  4명이 commit하면
      ▼
[세션 생성]             모임 확정 — 디스코드에 시간 다이제스트 알림
      │
[카카오톡]              사람들이 최종 시간을 말로 조정
```

**왜 1단계가 아니라 2단계인가?**
"이 날 시간 된다(가능)"와 "이 날 진짜 하자(참석의사)"는 다른 신호다. 가능한 사람 4명이
찍혔다고 바로 확정해버리면 성급하다 — 마음이 바뀔 수도, 시간이 안 맞을 수도 있다. 그래서
가능과 참석의사 사이에 "한 번 더 의도적으로 도장 찍는" 합의 비트를 끼웠다. 이게 성급한
확정을 막는다.

**핵심 숫자는 4 하나뿐.** 가능도 4명, commit도 4명. 그룹의 기존 룰("4명 모이면 성사")과
일치시켜 머릿속 모델을 단순하게 유지했다.

## 3. 아키텍처 한눈에 + 왜 그렇게 골랐나

| 층 | 선택 | 한 줄 이유 |
|---|---|---|
| 프레임워크 | **Next.js 풀스택** (프론트+백 한 덩어리) | 로직이 내 코드 한 곳에 있어 추적·디버깅이 쉬움 |
| 배포 | **Vercel** | Next.js의 고향, Cron 기본 제공 |
| DB | **Supabase (Postgres)** | 앱의 본질이 count/임계 → SQL이 홈그라운드 |
| DB 접근 | **Drizzle ORM** | SQL에 가깝고 타입세이프, 가벼움 (마법 적음) |
| 알림 | **Discord 웹훅** | 봇 호스팅 없이 HTTP POST 한 번이면 멘션 발사 |
| 리마인더 | **Vercel Cron** | 매일 9시 자동 실행 |

**왜 애초에 "서버"가 필요한가?** (순수 정적 React로 안 되는 이유)
세 가지가 서버 측 실행을 강제한다:
1. **공유 상태** — 친구들이 각자 찍은 걸 한 곳에 모아야 함 → DB
2. **임계 순간 웹훅** — 4번째 가능이 찍히는 *그 순간*을 감지해 디스코드로 쏴야 함 → write 시점 서버 로직
3. **매일 9시 리마인더** — 주기적으로 깨어나야 함 → cron

**왜 풀스택(BaaS 아님)?** 모든 로직("요청 → DB → 임계 체크 → 웹훅")이 내 코드 한 곳에 있어
위에서 아래로 읽힌다. BaaS(Firebase 등)는 그 로직이 플랫폼 트리거·함수 설정 안에 숨어서,
잘 돌 땐 마법 같지만 안 될 땐 어디가 문제인지 추적하기 어렵다. 배우는 입장에선 "내가 안 짠
마법"이 가장 위험하다.

**왜 Postgres(SQL)?** 이 앱이 하는 일의 99%는 "세기"다 — "6/20에 가능 찍은 사람 4명
넘었나?". SQL에선 `SELECT count(*) WHERE date='6/20'` 한 줄. NoSQL(Firestore)은 join도 count도
약해서 카운터를 손으로 관리하거나 컬렉션을 통째로 읽어 세야 한다. 임계가 핵심인 앱을 NoSQL로
짜면 계속 결을 거스른다.

## 4. 요청 하나가 시스템을 흐르는 모습 (가장 중요한 학습 포인트)

**시나리오: 철수가 6/20에 "가능"을 누른다.** 그러면 안에서 이런 일이 순서대로 일어난다.

1. 브라우저의 React 화면 → Next.js 서버로 "철수가 6/20 가능" 요청을 보낸다.
2. 서버가 `availabilities` 테이블에 `(철수, 6/20)` 한 줄을 넣는다.
3. 서버가 6/20의 가능 인원을 센다(`count`). 방금 4명이 됐나?
4. 됐고 **+** `notifications`에 "6/20 잠정 알림" 기록이 아직 없고 **+** `sessions`에 6/20이
   아직 없으면 → **디스코드 웹훅 발사**("🔥 6/20 잠정 성립") 후 `notifications`에 기록을 남긴다.
5. 서버가 결과를 돌려주고, 화면이 갱신된다.

**commit도 똑같은 패턴이다.** `commits`에 한 줄 넣고 → 6/20 commit 인원을 세고 → 4명이면
`sessions`에 6/20 행을 만들고(세션 생성) → 세션 생성 웹훅을 쏜다(각자 시간 + 겹치는 시간 포함).

이 "**삽입 → 세기 → 임계 넘으면 알림**" 패턴이 시스템 전체를 관통하는 한 가지 리듬이다.
이것만 이해하면 나머지는 변주다.

## 5. 데이터 모델 (테이블 6개)

```
members        id · name · discord_id                        ← 명단(Roster). 이름 + 디스코드 ID
availabilities member_id · date                              ← 가능 (날짜만). 한 사람이 한 날짜에 한 번
commits        member_id · date · time_start? · time_end?    ← 참석의사 (날짜 + 시간창)
sessions       id · date · created_at                        ← "세션이 생성된 사실" (날짜당 한 행)
topics         id · date · presenter_id · text               ← 발표자/주제 메모 (한 날에 여러 개 가능)
notifications  date · event_type · sent_at                   ← 웹훅 중복 발사 방지 기록
```

- `members`: 친구 명단. `discord_id`는 "그 4명만 콕 집어 멘션"하려고 미리 한 번 채워둔다.
- `availabilities`: "이 날 시간 됨"의 기록. 날짜만 있고 시간은 없다.
- `commits`: "이 날 진짜 하자"의 기록. 시간창(`time_start`, `time_end`)이 붙는다.
- `sessions`: 4명이 commit해서 모임이 확정된 날. **이 행의 존재 자체가 "세션이 있다"는 뜻.**
- `topics`: 누가 뭘 발표할지 대략 적는 메모. 선택이고, 한 세션에 여러 개일 수 있다.
- `notifications`: "이 날짜에 이 알림 이미 쐈다"를 기억해서 같은 알림이 두 번 나가지 않게 한다.

> `dates` 테이블은 따로 없다. 날짜는 그냥 날짜 값이라, 각 테이블이 직접 들고 있으면 충분하다.

## 6. 비자명한 모델링 결정 4개 (왜 이렇게 했나)

**① 세션은 "행으로 박고"(저장), 잠정 성립은 "그때그때 계산한다".**
세션은 한 번 생기면 안 깨진다(스티키). 그래서 반드시 `sessions` 행으로 영속화해야, 나중에
누가 commit을 빼도 행이 남아 세션이 유지된다. 반대로 잠정 성립은 깨질 수 있으니 저장하지
않고 "가능 4명 이상 **AND** 세션 없음"으로 필요할 때마다 계산한다.
→ 원칙: **저장은 비가역적인 것만. 계산할 수 있는 건 저장하지 않는다.**

**② 시간창은 nullable 두 칸으로 세 경우를 다 표현한다.**
`time_start`·`time_end` 둘 다 비어 있으면 = "시간 무관", 시작만 있고 끝이 비면 = "7시~"처럼
이후로 열림, 둘 다 있으면 = "7-9시" 같은 닫힌 창. 시간은 **자유 입력**이고 7시는 편의
프리셋 예시일 뿐, 강제 디폴트가 아니다.

**③ 웹훅은 멱등(idempotent)해야 한다.**
`notifications` 테이블이 "이 날짜에 잠정 알림 이미 쐈나?"를 기억한다. 없으면 5번째, 6번째
가능이 찍힐 때마다 "잠정 성립!"이 또 나간다. 그래서 쏘기 전에 "임계 넘었나 **AND** 아직 안
쐈나"를 둘 다 확인하고, 쏜 뒤 기록한다.

**④ Topic(발표자/주제)은 세션이 아니라 날짜에 매단다.**
잠정 단계(세션이 생기기 전)에도 "나 이거 발표할래"를 적을 수 있어야 하니까. 그 날짜가 나중에
세션이 되면 메모가 자연히 딸려온다.

## 7. 알림이 작동하는 법

- **플랫폼 분리**: 디스코드 = 기계가 말하는 곳(자동 알림). 카카오톡 = 사람이 말하는 곳(잡담,
  최종 시간 조정) + 링크 고정. 카톡엔 자동화가 일절 없다.
- **타겟 멘션**: `members.discord_id` 덕분에 "해당되는 사람만" 멘션할 수 있다. 잠정 성립은
  가능 찍은 사람들에게, 세션 생성·리마인더는 commit한 사람들에게.
- **늦은 합류(late join)**: 세션이 *이미 생성된* 날에 **새** 사람이 commit하면, 조용한(핑 없는)
  메시지로 "🙋 OO도 합류 (이제 N명) + 갱신된 겹치는 시간"을 알린다. 기존 commit한 사람의 시간
  *수정*은 알리지 않는다(새 사람만 트리거).
- **@everyone 절제**: 채널이 @everyone 폭격을 받으면 다들 알림을 꺼버린다 → 알림 시스템 자체가
  죽는다. 그래서 타겟 멘션을 기본으로 쓴다.
- **붕괴는 알리지 않는다**: 잠정 성립이 4→3명으로 깨져도 조용히 풀린다(노이즈 방지).
- **리마인더 = keep-alive 겸용**: 매일 9시 Vercel Cron이 "내일 세션 있나?" DB를 찌른다. 이게
  Supabase 무료 티어의 "7일 무활동 시 일시정지" 함정을 자동으로 막아준다.

## 8. 시간표(Timetable) UI

- 잠정 성립된 날짜를 누르면 **그 날 하루짜리** 시간표가 열린다 (날짜축 없는 when2meet).
- **열 = 사람, 세로축 = 시간.** 각자 자기 열에 가능 시간을 막대로 남긴다.
- **막대는 출력, 입력은 따로다.** MVP에선 숫자/프리셋으로 시간을 입력하면 앱이 그걸 예쁜
  막대로 그려준다. 막대를 직접 드래그하는 입력은 Phase 2(모바일 터치에서 까다로움).
- **추천 시간(Suggested Time)**: 4명 이상 겹치는 시간 행을 자동 하이라이트한다. 이건 "확정
  조건"이 아니라 "이 시간이면 다 되겠네"라는 제안일 뿐. 시간은 세션 생성을 막지 않는다
  (non-gating).

## 디자인 방향

**미니멀·깔끔한 애플 스타일.** 여백 넉넉히, 타이포 위주, 장식 최소, 색은 절제. 유틸리티
앱답게 "조용하고 정돈된" 인상. `frontend-design` 스킬로 이 방향을 잡으며 빌드한다.

디자인 투자가 가장 값하는 두 컴포넌트 (나머지 버튼·폼은 평범해도 됨):
- **캘린더 히트맵** — 인원이 색 농도로 차오르는 게 한눈에 읽혀야 한다("달아오르는" 느낌).
- **시간표 겹침 시각화** — 열=사람, 겹치는 시간대가 도드라지는 이 앱의 시그니처.

폴리시(`gstack-design-review`)와 컴포넌트 디테일 투자는 Phase 2.

## 9. 의도적으로 MVP에서 뺀 것 (Phase 2)

- **Program(활동 이벤트)** — 발표 말고 영화/TRPG/보드게임/술 같은 host-driven 모임. 8개월에
  2번뿐이라 희소. 흐름도 다르다(탑다운 초대 + 모객). 발표 세션이 완성된 뒤 따로 붙인다.
- **드래그 시간 입력** — 막대 직접 끌기. 데스크탑 polish.
- **최종 시간 앱 기록** — 카톡에서 정한 진짜 시간을 앱에 다시 적기. 지금은 앱이 모른다.
- **Realtime** — 남이 찍으면 내 화면에 실시간 반영. Supabase가 지원하지만 MVP엔 refetch면 충분.
- **모객 알림** — Program이 들어올 때 함께 검토.
- **어드민 계정 + 세션 수동 취소** — 생성된 세션을 취소하는 건 비가역·중요 결정이라, 아무나가
  아니라 어드민 계정(진근)만 가능해야 한다. 같은 어드민이 Roster 관리도 겸한다. (설계
  원칙: 더하는 행동은 누구나, 부수는 행동은 어드민만.)
- **지난 세션 이력 꾸미기 (발표 기록)** — MVP는 단순 리스트. "6/13 — 철수: TRPG" 식의 예쁜
  모임 일지/발표 기록으로 다듬는 건 Phase 2.
- **디자인 폴리시** — `gstack-design-review`로 만든 UI를 다듬고, 시그니처 컴포넌트(히트맵·시간표)에
  디테일 투자.
